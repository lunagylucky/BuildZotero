# 数据细节追问防守手册｜BuildZotero 面试兜底册

> **使用方式**：本册不替代 08–13，是**上台前 30 分钟**的最终核对表。每个数字/名词/例子配三层：
>
> - **L1 单点确认**：面试官随口一问的复述口径（10 秒内给结论）。
> - **L2 计算分解**：面试官深问"怎么算出来的"时的 30 秒拆解。
> - **L3 压力质疑**：面试官说"这个数是不是虚的"时的承认边界 + 反打方式。
>
> **核心原则**：承认不确定 + 给替代口径 + 把问题结构化，比假装精确更可信；**不新增未经事实底稿核对的数据**。

## 一、简历三行数字全部防守（L1/L2/L3）

> **事实底稿**：[[项目-BuildZotero-01简历撰写]]、[[项目-BuildZotero-06项目介绍梳理]]。以下所有答案必须与底稿一致，不随口扩大。

### 1.1｜"约 70% 链路耗时缩减"

- **L1（10 秒复述）**：相对于自己**优化前的同一条端到端链路**（从 PDF 入库到结构化标签可检索）的**估算**，不是行业横向对比，也不是单模型速度对比。
- **L2（30 秒拆解，4 项累加式）**：
  1. **多模型路由**：标签解构用 DeepSeek V3 批处理替代一律 GPT-4 级别长推理，单篇推理时间约 60% 的任务耗时减少（相对基线）。
  2. **批处理聚合**：100 篇论文从"逐篇开会话 × 100 次"合并为"1 次批处理 + 结构化解析"，重复解析 PDF 的成本一次完成。
  3. **二层检索结构化**：下游综述 / 引用不再每次重新读全文，直接从 Zotero 标签字段取结构化输入——前序代价摊到索引一次付清。
  4. **工作流打通**：Zotero → Obsidian → Word 的手动复制粘贴 / 格式重排工序取消。
- **L3（压力质疑承认 + 反打）**：
  - 承认：这是**估算量级**，不是严格 A/B 实验。采样路径约 10 批次，我自身操作、同一时段、同一模型配置。
  - 反打：**但链路层面的改动是可验证的**——路由策略、批处理、字段结构化这三件事本身可以复现；70% 是这三者叠加后的观察区间（60%–80%），对外选保守口径报 70%。
  - 替代口径：若追问严格性，直接说"单篇结构化处理从约 30 分钟降到 1 分钟以内"（详见 §1.2）——这是更具体可验证的指标。

---

### 1.2｜"单篇文献结构化处理约 30 分钟 → 约 1 分钟以内"

- **L1**：这里"30 分钟"是**人工打标签 + 人工整理变量 / 方法 / 理论**的历史经验耗时；"1 分钟以内"是**自动脚本跑完 8 维标签 + 写回 Zotero**的观察耗时。
- **L2（拆解 30 分钟）**：
  - 阅读摘要 + 扫读正文约 10 分钟；
  - 手工记录研究方法、样本、变量定义约 10 分钟；
  - 写入 Zotero 或 Excel 作为后续综述输入约 5–10 分钟。
- **L2（拆解 1 分钟）**：
  - PDF 全文抽取 + 分段 约 5–10 秒；
  - Prompt 请求 8 维标签 约 30–50 秒（取决于模型）；
  - JSON → Zotero 字段写回 约 5 秒；
  - 合计：视模型负载，约 45–70 秒波动，对外报"1 分钟以内"。
- **L3 压力质疑**：
  - 追问"30 分钟是不是你自己慢"：**对，是我自己研究生阶段读综述的典型速度**，不代表所有人；但核心改变不是"从慢人变快人"，而是"把**结构化标签化**这件事从人工 → 自动"，质量和可查性也变了。
  - 追问"1 分钟包不包括核对"：**不包括**。核对由人工做，白盒可改——这是"AI 给草案、人核结论"的流程设计，不是隐瞒。

---

### 1.3｜"标签准确率 80–90%"

- **L1**：这是**抽样自测区间**，不是第三方评估；按**模块 / 字段**差异较大，不是一刀切。
- **L2（按字段拆）**：
  - **主题（Item）、方法（sMeth）、样本（Sample）**：文本中**有明显明文**的字段，抽检准确率通常在 **90%+**。
  - **变量（A1–DV … A6-CV）**：需要文本理解 + 关系推断，典型 **80%–85%**。
  - **理论（Theory）**：最易出错，裸模型识别约 **60%–70%**；**加理论库匹配 + `Theory/new-` 标记后**，关键字段准确率可提到 **>95%**（底稿口径）。
  - **结论（Result）**：依赖对"主线结论 vs 次要发现"的区分，**约 75%–85%**。
  - **条目细节（V1–V5）**：依赖 Prompt 版本（V / V-R / V-Rnote），区间较宽 **70%–90%**。
- **L3 压力质疑**：
  - 追问"怎么抽检"：**分层抽样**——每批次每模块随机抽 10–20 条，我本人比对原文；不是跑在一个标准测试集上，这是**弱点**，承认。
  - 追问"有没有过拟合我自己的偏好"：**有**——我是同时设计 Prompt 和验收的人，这是真实偏差。改进方案：请课题组其他研究生做盲评。
  - 反打：**但关键字段（理论）有独立兜底机制**——理论库匹配 + `new-` 标记 + 白盒可改，单一字段错不伤整体。

---

### 1.4｜"万级文献架构能力 / 单次 200–300 篇输入"

- **L1**：**"万级"指架构能承载的条目数量级**，不是我真的跑了 1 万篇；**"200–300 篇一次输入"指治理后的结构化标签拼接**作为综述矩阵输入，**不是单窗口塞 200 篇全文**。
- **L2（为什么 200–300 篇能装进上下文）**：
  - 每篇经 P1 结构化后，实际用于综述的字段（主题 + 方法 + 样本 + 变量 + 结论）**压缩到 ~300–500 token**；
  - 200 篇 × 400 token ≈ **80k token**，加上 Prompt 模板与综述指令约 10k，**总 ~90k token**，在 Gemini 1M / 2M 上下文窗口内完全可行。
  - 对比"单窗口塞全文"：200 篇 × 6000 词 × 1.3 token/词 ≈ **150 万 token**，超出绝大多数模型上下文窗口，这就是竞品的瓶颈。
- **L3 压力质疑**：
  - 追问"万级真的跑过吗"：**没有连续跑 1 万篇**——但架构在 Zotero 数据库层面支持（Zotero 单库上限远超 10 万条目），P1 脚本是**逐条异步调用**，规模等比扩展。**真实验证**的最大一次是 200–300 篇量级综述任务。
  - 反打：**"万级架构能力"说的是工程边界**，不是"我跑了 1 万"；这是简历原句的保守表述。

---

### 1.5｜"清华、中科大等多个实验室私有化部署"

- **L1**：**私有化部署 = 脚本安装包 + Prompt / API 配置本地化 + 1–2 次远程或现场培训**，不是 SaaS 级别的 On-Prem 部署。
- **L2（部署流程 5 步）**：
  1. **环境对齐**：Zotero 版本、Obsidian 版本、Zotero GPT Pro 插件版本核对；
  2. **API 密钥配置**：实验室自己的 DeepSeek / Gemini / Claude API Key 本地注入，密钥不经我方中转；
  3. **脚本下载**：7 模块 JS 脚本本地化；
  4. **Prompt 模板**：按实验室学科（经管 / 信息 / 材料等）微调 8 维标签说明文案；
  5. **培训 + 回访**：2 小时使用培训 + 1–2 次远程答疑，收集问题进迭代池。
- **L3 压力质疑**：
  - 追问"具体哪个实验室、PI 叫什么"：**涉及隐私与课题组敏感**，我倾向不公开具体 PI 姓名；**但可以确认学校与学科方向**（清华 + 中科大，经管 / 信息方向）——和简历底稿一致。
  - 追问"几个人在用"：**种子用户约 10–20 人量级**（不夸大"百人团队"），以每个实验室 3–5 人估算。
  - 反打：**我关心的是「真实在用」而不是「用户数」**——回访里最让我有成就感的是"综述从两周压到几天"这种反馈。

---

### 1.6｜"盈利 10 万元"

- **L1**：**项目总收入约 10 万元人民币级别**，来源为私有化部署服务费 / 课题组按需咨询费；**不是 SaaS 订阅规模**，**不是持续 ARR**。
- **L2（构成拆解）**：
  - 部署服务费：按课题组规模一次性计费（几千到一万级别）；
  - 培训 / 定制 Prompt：按工时或包月计费；
  - 合计**累计**约 10 万元——不是月收入，不是毛利。
- **L3 压力质疑**：
  - 追问"有没有合同 / 发票可出示"：部分为私下课题组服务 / 开源赞助形式，**正式合同较少**；对外口径应保守，不主动扩大商业化描述。
  - 追问"毛利率"：**扣除 API 调用 + 时间成本后实际毛利有限**，这个项目更多是"学术价值 + 方法论沉淀"，商业化规模不强。
  - **若 HR 追问过深**：改口径为"项目验证了付费意愿，但未做商业化系统建设"——与 [[项目-BuildZotero-02-任务主线与业务面试问答]] §Q9 一致。

---

### 1.7｜"119 个版本文件 / 300+ Prompt / 33 个功能脚本 / 52 个功能点"

- **L1**：
  - **119 = 仓库中 P0–P6 七个模块累计的版本迭代 `.js` 文件数**（同一功能多版本算多）；
  - **300+ = Prompt 片段 / 模板 / 脚本颗粒累计数量**，含迭代中替换的旧版本；
  - **33 = 独立功能数量**（不计版本变体）；
  - **52 = 功能点**（含版本变体与子功能）。
- **L2（模块拆）**：
  - P0 基础操作：4 个功能；P1 文献标签解构：7 个功能；P2 统计：3 个功能；P3 筛选：4 个功能；P4 综述：5 个功能；P5 引用：1 个功能；P6 交互：9 个功能。合计 **33**，加版本变体共 **52**。
- **L3 压力质疑**：
  - 追问"119 个版本是不是堆量"：**不是重复复制**——每个版本对应一次 Prompt 调整或 Bug 修复或模型路由策略变化；**用 `V / V-R / V-Rnote` 版本约定管理**（见 §2.1）。
  - 反打：**版本多恰好说明迭代认真**——每次改都保留可回滚点。

| **P1 文献标签解构** | 核心 | 8 维标签提取、Prompt 模板化、Zotero 字段写回、V/V-R/V-Rnote 三版本、理论库匹配 | 壁垒 3（输出不稳定）+ 壁垒 2（幻觉） |
| **P2 统计看板** | 全局透视 | 按主题/方法/理论分布统计、时间趋势、作者/期刊分布 | 壁垒"全局态势感知模糊" |

| **P5 引用匹配** | 写作辅助 | 精准匹配（1–2 篇）、穷尽匹配（所有相关）、多格式输出（APA / GB/T 7714 / 自定义） | 工作流断裂（壁垒 4） |
| **P6 交互问答** | 对话层 | 选中文本问答、PDF 问答、全文问答、笔记问答、知识库问答、剪贴板问答、标注问答、元数据问答、文件上传问答（9 种） | 综合交互，工作流闭环 |

**面试话术抓手**：*P0 打地基 → P1 结构化 → P2 看全貌 → P3 查细节 → P4 批量产出 → P5 引用 → P6 问答。像"先装好柜子、再给每本书贴标签、再站远看全图、再精准取书、再写书评、再引用、再问答"——顺序即架构。*

### 2.2｜8 维标签体系（具体字段名 + 举例）

| 维度 | Zotero 字段名 | 内容示例 | 为什么单独抽 |
|---|---|---|---|
| **主题** | `Item/xxx` | `Item/ESG-披露-分析师预测` | 学科 + 核心构念，用于 P2 分布统计 |
| **方法** | `sMeth/xxx` | `sMeth/DID-双重差分`、`sMeth/OLS` | 用于 P3 按方法筛选、P4 按方法分组综述 |
| **样本** | `Sample/xxx` | `Sample/CN-A股-2010-2023` | 复现性必备 |
| **理论** | `Theory/xxx` 或 `Theory/new-xxx` | `Theory/信号传递`、`Theory/new-注意力经济学` | 最容易幻觉，有理论库兜底 |
| **结论** | `Result/xxx` | `Result/披露质量提高分析师准确率` | 一句话主线结论 |
| **变量** | `A1-DV`（因变量）到 `A6-CV`（控制变量） | `A1-DV/分析师预测偏差`、`A2-IV/ESG 披露评分` | 研究设计骨架，P4 矩阵的列 |
| **条目细节** | `V1-def`（定义）、`V2-mea`（测量）、`V3-con`（结论）、`V4-fut`（未来研究）、`V5-sor`（来源） | `V4-fut/建议拓展至港股` | 支持 P4 研究议程识别 |
| **版本** | `V`（标准版）/ `V-R`（重跑版）/ `V-Rnote`（笔记优先版） | 同一条目可有多个版本，便于 A/B 比较 Prompt | 可回滚 |

**追问防守**：
- "为什么把变量分成 A1–A6 六类"：A1 = DV（因变量）、A2 = IV（自变量）、A3 = MO（调节）、A4 = ME（中介）、A5 = INV（工具变量）、A6 = CV（控制变量）——**社科实证研究设计的标准骨架**，少一类综述矩阵就漏。
- "条目细节 V1–V5 怎么用"：针对同一构念的**纵向追问**——从定义（V1）→ 测量方法（V2）→ 结论（V3）→ 未来方向（V4）→ 来源（V5），支持 P4 综述的"构念深挖"模式。

### 2.3｜11 维分析框架完整列名（综述矩阵的列骨架）

> 面试不必死背 11 个，但要能**举 4–5 个**并说出"为什么是这 11 个、用途"。

1. **研究背景**（Background）—— 社会 / 学科 / 政策背景
2. **构念定义**（Construct）—— 关键概念如何定义
3. **测量方法**（Measure）—— 量表 / 客观指标 / 文本分析
4. **理论视角**（Theory）—— 主导理论 / 替代解释
5. **对象层次**（Level）—— 个体 / 组织 / 国家
6. **研究情境**（Context）—— 国家 / 行业 / 时间
7. **研究方法**（Method）—— 实证 / 案例 / 实验 / 元分析
8. **前因**（Antecedent）—— IV 和其前置条件
9. **后果**（Consequence）—— DV 和其后续影响
10. **中介**（Mediator）—— ME
11. **调节**（Moderator）—— MO

**追问防守**：
- "为什么是 11 个不是 10 或 12"：**综述做多了你会发现**——少了中介 / 调节综述显得机械；少了构念定义综述读不懂；11 维是**能跨学科复用**的最小完备集。
- "跨学科适用吗"：社科管理学完全适用；自然科学可能需要改 2–3 列（如把"理论视角"改为"机制"、"构念"改为"现象"），**列名不同、列位数相近**。

### 2.4｜Prompt 三层分层（带片段示例）

#### 第一层：角色与规范注入（示例片段）

```
你是一个严谨的学术文献结构化分析员。
领域定义：
- 因变量(DV) = 被解释变量，量化可测；
- 自变量(IV) = 核心解释变量；
- 控制变量(CV) = 其他影响 DV 但非研究焦点的变量；
- 理论(Theory) = 解释 IV→DV 机制的成熟命题；
- 若你对某个理论命名不确定，请在标签前加 `Theory/new-` 前缀。
```

#### 第二层：任务与格式约束（示例片段）

```
请从给定文献 PDF 全文中抽取以下 8 维标签，严格按 JSON 输出：
{
  "Item": "xxx",
  "sMeth": "xxx",
  "Sample": "xxx",
  "Theory": ["Theory/xxx 或 Theory/new-xxx"],
  "Result": "xxx",
  "Variables": {"A1-DV": "...", "A2-IV": "...", ...},
  "Details": {"V1-def": "...", "V2-mea": "...", ...}
}
要求：
- Theory 字段必须先尝试与理论库匹配，未命中才用 `new-` 前缀；
- Result 字段限一句话≤50字，描述主线结论；
- 变量字段空缺留空字符串，不编造。
```

#### 第三层：质量控制（示例片段）

```
质量检查清单（内部执行，不输出）：
1. Theory 是否来自理论库？若否是否加 `new-`？
2. A1-DV 和 A2-IV 是否都来自文献明文，非推断？
3. JSON 是否 parseable？
若任一项不满足，请重新检查上下文并修正输出。
```

**追问防守**：
- "你怎么保证 JSON 不破"：**不靠模型承诺，靠解析失败自动重试**——下游 JS 用 `try/catch` 解析，失败就降温度重跑一次（最多 2 次），再失败标 `parse-fail` 人工处理。
- "Prompt 迭代了多少轮"：**P1 的主 Prompt 至少 30+ 轮**，每轮改动记录在版本文件里（见 §1.7 的 119 个版本）。

**"降本约 73%" 的拆解（口径与 [[项目-BuildZotero-13专业知识文档-AI应用与产品经理]] 一致）**：
- 基线：全部任务用 GPT-4 / Claude Opus 级别模型，**单篇标签 + 综述 + 问答平均 API 成本约 $0.15–$0.25 / 篇**。
- 优化后：按任务路由到 DeepSeek V3（批处理路径 ~$0.01/篇）+ Gemini（关键路径 ~$0.05/篇）+ 缓存命中（约 30% 请求），**平均成本约 $0.03–$0.07 / 篇**。
- **节省 ≈ (0.20 − 0.05) / 0.20 ≈ 75%**，对外保守口径报 **~73%**。

**追问防守**：
- "这个基线是不是你挑的最贵模型"：**承认**——基线假设"一刀切用最强模型"，这是最保守的对比；真实企业里可能基线本身就做了部分分流，所以 73% 是**相对"无路由"的上限估计**。
- "缓存命中 30% 怎么测的"：**相同 PDF / 相同 Prompt 的重复请求率**，在迭代期特别高（同一篇反复调 Prompt）；量产期会降到 10%–15%。

## 三、迭代故事三则（行为面与追问双用）

### 故事 1｜理论库循环机制的由来（真实迭代）

- **背景**：某次用户反馈"P1 给一篇研究披露的论文打了 `Theory/注意力资本理论`，但这个理论并不存在（模型幻觉命名）"。
- **我的处理**：
  1. 复现问题：换 3 次模型，2 次给 `Theory/注意力资本理论`、1 次给 `Theory/信号理论`——**证实不稳定**；
  2. 定义问题：这不是"模型好坏"，是"没有一个人能穷尽所有理论"——**问题在框架不在模型**；
  3. 设计机制：引入**标准理论库 + `Theory/new-` 前缀**——模型若匹配到库中理论直接用，匹配不上加 `new-` 前缀**醒目标记待人工核**；
  4. 关键字段准确率目标从 **60%–70% 提升到 >95%**（底稿口径）。
- **可迁移**：任何"AI + 专业知识库"场景都有类似结构——案例库、方法库、合规库都可套用。

### 故事 2｜"万级文献"的真实规模验证

- **背景**：某实验室反馈"把 800 篇文献一起跑 P4 综述，脚本卡住了"。
- **我的处理**：
  1. **定位**：不是 AI 模型问题，是**我脚本的并发控制不当**——一次 800 个 API 请求触发 Rate Limit，累积失败重试把内存吃满；
  2. **修复**：引入**并发池 + 退避重试**（最多 5 并发、指数退避、失败重试 3 次），任务队列化；
  3. **结果**：800 篇跑通，耗时约 25 分钟（含重试），之后此机制成为所有批处理模块的标配。
- **可迁移**：任何"批量 × AI API"场景都得做限流/重试。这是我从这个项目里**最具体的工程收获**。

### 故事 3｜二层检索的由来

- **背景**：某用户反馈"AI 打了一批标签，但我自己的原标签体系被污染了（`主题/ESG` 和 `Item/ESG-披露` 混在一起）"。
- **我的处理**：
  1. **诊断**：用户的"主题"是**个人长期整理的认知地图**，AI 字段是**自动提取的研究骨架**——**混在一起就是双输**；
  2. **设计**：**二层检索**——AI 字段用 `Item/`、`sMeth/`、`Theory/` 等命名空间前缀与用户原标签隔离；用户原标签**完整保留，不受 AI 修改**；
  3. **结果**：用户可随时**删除 AI 字段而不伤自己的体系**；P3 检索可选择"只查 AI 字段"、"只查用户字段"、"都查"。
- **可迁移**：**数据治理的核心永远是「分清谁是 source of truth」**。

## 四、常见高杀伤追问 20 问（L1/L2/L3 简版）

### Q1｜你不是 CS 科班，怎么保证代码质量？

- L1：我做**设计 + 集成**，不造复杂算法；代码强度主要在 Prompt 工程与 API 编排。
- L2：**Python 数据分析训练迁移到 Zotero 的 JS 语境**；用版本化（119 版）+ 模块切分（7 模块）控制可维护性；复杂算法交给成熟模型。
- L3：承认**如果要做底层基础设施（如向量数据库、推理引擎）我不合适**；但做**应用层产品与数据治理**，我完全胜任。

### Q2｜ 你说的"用户"到底是谁？有多少人？

- L1：**科研端种子用户 10–20 人量级**，分布在清华、中科大等实验室；不是 C 端用户规模。
- L2：按实验室维度算：**至少 2–3 个实验室真实部署**，每个 3–5 人使用；加上开源脚本未登记使用者，量级在 10–30 人。
- L3：承认"用户数"不是我最在意的指标；我在意的是**真实使用频次 + 留存反馈**（如"综述时间从两周压到几天"这类定性反馈）。

### Q3｜AI 幻觉你真的解决了吗？

- L1：**没有"解决"，只是"受控"**——用白盒字段 + 理论库 + 证据锚点把幻觉的破坏面缩到可验收范围。
- L2（五层防御，详见 [[项目-BuildZotero-13专业知识文档-AI应用与产品经理]]）：①结构化输入；②理论库约束；③锚点溯源；④人机协同白盒；⑤LLM-as-Judge 兜底（设计中）。
- L3：承认**创造性任务（综述）的幻觉不可能 0**；但**可验证的字段类输出（标签、引用）**幻觉可压缩到个位数百分比。

### Q4｜没有后端、没有数据库，算什么"系统"？

- L1：**Zotero 本身就是数据库（SQLite + 嵌入式），我的系统是在其上做解构与编排**——**选对生态 > 造轮子**。
- L2：后端层面的事（存储、索引、全文搜索）Zotero 已解决；我的价值在**数据流设计 + AI 编排 + Prompt 工程**。
- L3：如果要做独立 SaaS，**我知道会新增什么**（多租户、鉴权、数据隔离、SLA），但当下不是项目目标。

### Q5｜Zotero GPT Pro 是别人写的，你只是调用？

- L1：**对，这是我选生态的结果**——不重复造底层 Zotero 改写能力。
- L2：我的工作在**插件之上**——脚本架构、Prompt 体系、API 编排、8 维标签规范、理论库机制、二层检索设计、综述矩阵、引用匹配、工作流打通。**这些都不是 Zotero GPT Pro 自带的**。
- L3：承认**如果 Zotero GPT Pro 大改 API 我要跟改**；但这也是生态型产品的普遍成本，Obsidian 插件生态同理。

### Q6｜你怎么证明标签准确率 80–90%？

- L1：**抽样自测**，每批次每字段随机抽 10–20 条，我本人核对原文。
- L2（分字段）：主题/方法/样本 90%+；变量 80%–85%；理论裸模型 60%–70%，加库匹配后关键字段 >95%；结论 75%–85%。
- L3：承认**样本量小 + 我自己评**，有偏；改进方向是请其他研究生盲评或建立金标准测试集。

### Q7｜ 你的"200–300 篇"是不是单窗口塞全文？

- L1：**不是**。是**治理后的结构化标签拼接输入**，200 篇 × 400 token ≈ 80k token，Gemini 1M 上下文内。
- L2：**裸全文 200 篇 ≈ 150 万 token**，超出大多数模型窗口——这正是竞品做不到的瓶颈，而我方**靠 P1 把输入压缩 10–30 倍**。
- L3：承认**综述质量与"压缩率"权衡**——过度压缩会丢细节；所以我设 V1–V5 版本，综述不够细时可拉 V-R 重跑。

### Q8｜"知识资产化率 >90%"是什么意思？

- L1：指**已入库文献中至少有 8 维标签里 7 维完成自动抽取**的比例。
- L2（计算）：某次 500 篇批处理，完成 8 维齐全的约 420 篇（84%），7 维齐全（容忍 1 维缺失）约 460 篇（92%）——**对外口径 "7/8 维完成度 >90%"**。
- L3：承认**这个指标是我自己定义的**，不是行业标准；面试如追问严格性可改口径："完成至少 7 维标签的条目占比"。

### Q9｜如果模型 API 断掉，你的产品还能用吗？

- L1：**基础功能能用**（P0 清理、P2 统计、P3 基于已有标签的检索）；**AI 重度功能不能用**（P1 解构、P4 综述、P6 问答）。
- L2：**这是"AI 增强型产品"的普遍风险**——我没有做本地小模型兜底，但做了**Prompt 与 API 配置层**解耦，实验室可切换到任意支持 OpenAI 格式的 API。
- L3：承认**高可用不是当前优先级**；如果要企业级部署需要再加本地推理兜底（如 Llama 3 70B 本地化）。

### Q10｜做到今天最后悔什么？

- L1：**没更早做可量化用户实验**——前期迭代过度依赖自身痛点。
- L2：如果重来，我会：①一开始就写**量化指标 SOP**（准确率、耗时、成本）；②做**盲评数据集**；③更早做**A/B 对比实验**（同一批文献新旧流程跑一遍）。
- L3：这也是为什么我**看重数据分析能力**——PM 要用数据说服团队，不靠感觉。

### Q11｜为什么是 Zotero 不是 Endnote / Mendeley？

- L1：**Zotero 开源 + 可扩展 + 有 JS Connector** ——其他商业软件不让改。
- L2：Zotero 的 SQLite 底层 + 插件生态允许我写入任意字段；Mendeley 2020 年后逐步关闭 API；Endnote 是黑盒。
- L3：**选 Zotero 是路径选择而非技术判断**——任何有 API 的开源文献管理器都可以，但 Zotero 生态最成熟。

### Q12｜你怎么处理版权与论文全文外发？

- L1：**本地优先 + 最小外发 + 课题组安全偏好对齐**——默认只发摘要 + 关键段落，不发全文，除非用户明确允许。
- L2（部署细节）：API Key 由实验室自持，请求不经我方中转；可配置"不外发"模式，走本地摘要抽取。
- L3：承认**真正的零外发需要本地模型**，这是未来方向；当前项目是"合理外发 + 透明记录"。

### Q13｜多模型路由你写死了吗？

- L1：**没有写死**，是基于**配置文件 + 任务标签**的分派，实验室可自定义"哪个任务走哪个模型"。
- L2：配置层结构类似 `taskType: modelName` 映射表；脚本读取配置后选择 API 路径；未来可改成基于质量 / 成本指标的动态选择。
- L3：承认当前是**规则路由**，不是**学习路由**；想做动态选择需要 LLM-as-Judge + 多臂老虎机，工程量大。

### Q14｜竞品如果抄你这套标签体系怎么办？

- L1：**标签体系本身是领域知识的开放表达**，抄不走护城河——真正难的是"标签 + 理论库 + 白盒 + 工作流"一体化。
- L2：护城河在**三层**：①Prompt 工程经验（迭代 30+ 轮才能稳）；②与 Zotero / Obsidian 工作流的深度绑定；③私有化部署的信任关系。
- L3：承认**AI 应用层同质化风险高**——长期护城河必须往**垂直领域数据资产 + 生态合作**走，这是我未来会做的方向。

### Q15｜你说"三阶段 Agent 架构"到底是哪三阶段？

- L1：**结构化 → 检索与生产 → 交互反馈**三阶段（与 P1→P3/P4→P6 对应）。
- L2：①**感知与结构化**（P0/P1）：把非结构化 PDF 转成可计算的标签；②**检索与生产**（P2/P3/P4/P5）：在结构化资产上做查询、综述、引用；③**交互与反馈**（P6）：对话式问答 + 锚点溯源 + 用户白盒修改。
- L3：承认这**不是严格学术意义上的"Agent"（带规划与工具调用循环）**；更接近"多阶段 Pipeline + 人机协同"，名字里的 Agent 是简历对外用词，我不回避这一点。

### Q16｜你对 Agentic RAG / GraphRAG 怎么看？

- L1：**我做的是介于 Naive RAG 与 Advanced RAG 之间**——结构化标签是一种"预索引的图 / 语义网络"原型，但不是严格的 GraphRAG。
- L2：GraphRAG 适合**关系强、实体多、跨文档**的任务（如法律案例链）；我的场景更像**每篇文献独立、主要是字段维度的矩阵化**。
- L3：承认**如果做跨文献的因果链追踪、理论演化追踪**，必须往 GraphRAG 走；这是下一版潜在方向。

### Q17｜"清华 / 中科大"能不能说具体名字？

- L1：**学校可说，具体实验室 PI 不说**——课题组敏感偏好。
- L2：如果面试官确需验证，**可提供联系人邮箱（经对方同意）**，不走简历公开路径。
- L3：这也是我**尊重用户隐私**的职业习惯——产品里也是本地优先、最小外发。

### Q18｜你这个项目能 scale 吗？

- L1：**Pipeline 能 scale（架构是异步 + 模块化）**，**商业化目前不 scale（高度依赖私有化部署 + 培训）**。
- L2：工程侧 scale 的瓶颈在**并发与缓存**——已有初版（5 并发、缓存命中 30%）；商业 scale 的瓶颈在**销售 + 交付**，我一个人做不了。
- L3：承认**项目定位是"科研向 + 开源脚本 + 少量私有化"**，不是 VC 级 SaaS；这是我如实的 positioning。

### Q19｜你怎么看大模型厂商把你的场景吃掉？

- L1：**大概率会吃掉**，这是 AI 应用层的普遍风险。
- L2（三层应对）：①**往垂直深挖**——领域知识、理论库、学科惯例是壁垒；②**往工作流嵌入**——Zotero / Obsidian / Cursor 的用户迁移成本高；③**往数据资产沉淀**——客户数据不回模型厂商。
- L3：长期看，**AI Agent 层通用化一定挤压中间件**；我对"只做薄壳 wrapper"的产品保持警觉，这也是为什么我强调"结构化 + 白盒 + 工作流"这三条——**这是 wrapper 不具备的价值**。

### Q20｜你下次做 AI 产品会先做什么？

- L1：**先定义可验证指标 + 做最小 A/B 实验 + 再扩大**。
- L2：按 [[项目-BuildZotero-13专业知识文档-AI应用与产品经理]] §12 的"15 题自检清单"走一遍；把**离线评估、在线评估、LLM-as-Judge、人评**四类都规划进来。
- L3：**做 AI 产品最大的坑是"demo 很酷，上生产翻车"**——这次项目我靠白盒 + 理论库 + 锚点三重兜底扛住了，下次会做得更系统。

## 五、压力测试场景模板（背到能应激反应）

### 场景 A｜面试官说"你这些数字我都不信"

**不推荐**：辩解"我真的测过"。
**推荐**：承认 + 结构化 + 反打。

> *"您的怀疑合理。这些数字有三种可信度层次：简历上的'70%'是我的估算，面试我也不会报 85% 或 55%——因为没法严格复现实验。但中间数字如'单篇 30 分钟→1 分钟以内'可复现，我可以现场演示。最硬的是架构层面的改动——多模型路由、批处理、结构化输入——这些不需要信我，代码和 Prompt 结构本身就是证据。您想从哪一层往下看？"*

### 场景 B｜面试官打断你，"别讲项目了，讲一个你失败的例子"

**推荐**：立即切到 §三故事 2（并发压垮）。

> *"最直接的失败是——某实验室把 800 篇一起跑，我的脚本卡了 10 分钟然后崩了。根因是我没做并发控制，800 并发请求把内存打爆。改法：引入并发池 + 指数退避。这次教训让我之后所有批处理模块都必须先过限流检查。"*

### 场景 C｜面试官质疑"这听起来像 ChatGPT 都能做"

**推荐**：用一句话反差 + 具体差异。

> *"如果您问 ChatGPT'请把我 Zotero 里 200 篇文献按方法分类做综述矩阵'——它做不到，因为①它不知道您的 Zotero 库；②它吃不下 200 篇全文；③它没法把结果写回 Zotero 字段供您半年后检索。BuildZotero 解决的是'AI + 本地数据 + 工作流'的三角，不是单点对话。"*

### 场景 D｜面试官说"你这个项目太小了"

**推荐**：承认规模 + 强调能力 + 岗位对齐。

> *"项目商业规模确实不大——10 万元级别，非 SaaS。但项目的价值在**能力展示**：需求归类（30+ 抱怨 → 12 瓶颈 → 4 壁垒）、架构设计（7 模块 52 功能点）、数据治理（8 维 + 11 维）、工程落地（部署 + 培训）——这些是贵司这个岗位要做的事。您愿意的话我可以把其中任何一层展开。"*

### 场景 E｜面试官直接问"下一步你打算怎么做"（考 vision）

**推荐**：不讲"要估值一亿"，讲**3–6 个月可验证的动作**。

> *"三件事：①把'7 模块自定义组合'变成可配置工作流（now→next），让每个实验室不用我亲自配就能适配；②引入 LLM-as-Judge 做在线质量监控（next→later）；③往相邻学科扩（法律、医学案例库），方法论 100% 可迁移。背后的理念不变——**结构化 + 白盒 + 工作流嵌入**。"*
