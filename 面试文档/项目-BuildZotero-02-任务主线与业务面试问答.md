## 一、开场话术（30 秒 + 2 分钟）

### 30 秒版本

我在 BuildZotero 里主要做三件事：第一，把非结构化论文通过 Prompt 与标注体系落成可改、可检、可复用的结构化标签；第二，用二层检索、统计看板和多维筛选解决「记得读过但找不到」；第三，用综述矩阵、引用匹配和带锚点的交互问答打通 Zotero—Obsidian—Cursor 工作流。我的核心价值是知识工程：让 AI 产出可复核、可追溯，而不是一次性会话。

### 2 分钟 STARR

**S**：科研中文献规模大、工具链割裂，通用 AI 读 PDF 往往是点状会话，知识难沉淀，综述与引用还要大量手工。  
**T**：在真实课题组场景里落地一套知识工程方案：结构化索引、白盒维护、与工作流集成，并可私有化交付。  
**A**：科研需求调研与竞品对照；设计 7 模块（P0–P6）与 8 维标签 + 11 维分析框架；JavaScript + API 编排 + 多模型路由；沉淀 PRD/架构/路线图等产品文档；在高校实验室做部署与培训。  
**R**：形成可支撑万级文库、并以结构化方式组织 200–300 篇量级输入做矩阵化产出的链路；引用与对话强调证据锚点；在清华、中科大等多个实验室落地使用。  
**R（反思）**：难点不在「会不会调模型」，而在数据治理与人机协同——AI 给草案，最终结构与证据必须能被人核对和改写。

### 三条核心贡献（一句话）

1. **数据治理**：非结构化文献 → 结构化标签 + 理论库循环更新 + 白盒修改。  
2. **检索与规模化生产**：二层检索、全景统计、「先治理、后生产」的综述/引用/对话输入策略。  
3. **工程化与落地**：多模型路由、脚本化交付、私有化部署与培训。

### 简历三行对齐（与 01 一致，面试勿漂）

**BuildZotero｜学术文献 AI 工具｜产品设计者 & 开发者｜被清华、中科大多个实验室采纳；盈利 10 万元｜中国，合肥｜2024.12–2026.1**

- **产品架构**：系统化调研科研群体 AI 应用需求，按科研流程解构 12 类瓶颈并输出 PRD；设计 Agent 三阶段架构与 7 大模块，将非结构化论文转化为结构标签。  
- **代理编排**：构建 7 模块异步代理框架，支撑万级文献自动化结构与交互系统；设计二层检索逻辑优化 RAG 精度与上下文限制，建立白盒化证据溯源链路。  
- **集成部署**：基于成本与质量指标的模型路由策略，打通跨工具链工作流并缩减约 70% 链路耗时；完成清华、中科大实验室私有化部署及使用培训。

## 二、考核点与回答原则

**面试官总评逻辑（产品 / AI / 知识工程岗通用）**

- 先看**真实性**：本人承担的设计、开发、部署动作与边界；数字口径是否稳定。  
- 再看**专业性**：是否具备「问题定义 → 架构 → 数据规范 → 验证指标」闭环，而非堆模型名。  
- 再看**协作与表达**：能否说清模块依赖与取舍（质量、成本、时延）。  
- 最后看**可迁移性**：能否抽象为「知识资产化」方法，并承认局限。

**简历高风险提问点（须在回答中自带解释口径）**

- 「万级文献」「200–300 篇一次输入」：区分**架构能力/真实任务口径**与「单窗口塞全文」；说明结构化 + 分阶段批处理。  
- 「70%」「80–90%」等：**相对基线/抽检**，备 1–2 种验证说法。  
- 「多个实验室采纳」：落实到部署、培训、回访，避免空泛。  
- 「盈利 10 万元」：与 01 一致方可说；被追问要能说明模式，否则改简历。  
- 「300+ 脚本 / 119 个版本文件」：**版本迭代与模块切分**导致文件多，不等于 300 个无关脚本。

**简历原句防守口径（不改 01，只解释 01）**

| 简历原表述 | 面试解释口径 |
| --- | --- |
| 缩减约 70% 链路耗时 | 相对优化前同一路径的端到端耗时 estimate；可拆成模型路由、批处理、避免重复解析。 |
| 标签准确率 80–90% | 迭代区间，抽检或对比修订；理论等关键字段靠理论库 + 白盒兜底。 |
| 检索查准率等 | 设计目标或自测口径；说明样本与局限。 |
| 被多个实验室采纳 | 私有化部署、培训与回访的具体叙述。 |
| 独立设计/开发 | 产品设计与主路径脚本为本人；第三方插件、API 边界说明清楚。 |

**真实性快检（面试官视角）**

- 数字能否说出**定义、样本、时段**？  
- 能否区分「我产出」与「课题组/用户反馈」？  
- 30 秒内能否画清 **PDF → 标签 → 综述/问答** 数据流？

**外部校准依据（可选）**

- RAG：先治理索引再生成，降低幻觉与无效上下文（行业共识）。  
- 人机协同：关键字段人工可改、可复核（HITL）。

### 主线 A：需求洞察、产品形态与竞品差异

#### **你做了什么**

- 从科研流程归纳痛点与竞品盲区，确立「结构化、白盒化、流转自动化」；交付物为 **Zotero + Obsidian + Cursor 生态上的脚本 + Prompt + API 编排 + 文档**，非独立封闭式 App。  
- 结构化调研与归类，产出 PRD；7 模块、52 功能点（对外文档份数在 8/15 等口径中选**一种说法**与 06 对齐）。  
- 明确与 Kimi、ChatPDF、NotebookLM、Elicit 等的能力边界。

#### **面试官提问 1**：为什么不做独立产品 / 你为什么押 Zotero 这条生态？  
**考核点**：是否理解科研用户的工具链、迁移成本与可维护性，而非「会做网站就上网站」。  
**高分标准**：用**工作流嵌入成本**和**白盒可控**解释，不贬低其他形态。  
**面试回答（总分）**：  
总：我选的是「站在文献库与笔记生态上增强」，而不是重复造文献管理或聊天界面。  
分：第一，研究者的主数据在 Zotero，增量价值应写回条目与标签；第二，独立 App 难覆盖各实验室千差万别的字段与保密要求；第三，脚本 + 插件形态便于课题组按需裁剪与私有化。

#### **面试官提问 2**：你和市面上 AI 读 PDF 工具的本质区别是什么？  
**考核点**：是否说清「点状会话 vs 永久资产」「黑盒 vs 白盒」范式，而不是功能罗列。  
**高分标准**：对比交互逻辑 + 可检索性 + 工作流，可辅以下表心智。  
**面试回答（总分）**：  
总：一句话是「结构化索引 + 多维检索 + 工作流嵌入」，不是单次对话体验。  
分：第一，通用工具多为点状会话，知识难沉淀；我把结果写入 Zotero 成为可检索资产。第二，检索从关键词升级为「变量 + 方法 + 样本」等组合。第三，输出可带锚点回链，配合白盒修改，抑制幻觉与不可追溯。  

| 维度 | 通用 AI 读 PDF | BuildZotero |
| --- | --- | --- |
| 交互 | 会话结束难沉淀 | 结构化索引进文库 |
| 检索 | 关键词为主 | 多维逻辑组合 |
| 规模 | 受上下文约束大 | 结构化 + 批处理组织输入 |
| 维护 | 黑盒难改 | 白盒可改、可版本 |
| 工作流 | 独立工具 | Zotero + Obsidian + Cursor 协同 |

#### **面试官提问 3**：需求你怎么发现的？调研过程能否 STAR 一说？  
**考核点**：从个人痛感到共性需求，方法是否结构化（访谈、归类、竞品、优先级）。  
**高分标准**：STAR 完整；痛点可归因到「质量不确定、检索单一、难积累、工作流割裂」等可检验表述。  
**面试回答（总分）**：  
总：从自己的综述瓶颈出发，再验证是否课题组共性。  
分：**S** 读过找不到、写综述对不上；**T** 判断值不值得做系统性方案；**A** 自身体验提炼 → 访谈/归类 → 竞品边界 → 重要性–紧迫性排序；**R** 形成 PRD 与模块路线。Hooks：可引面试官问「7 模块」或「理论库」。

### 主线 B：结构化抽取与检索（P0–P3）

#### **你做了什么**

- **P1**：8 维标签 + 11 维分析协议，Prompt 映射入 Zotero；V / V-R / V-Rnote；理论库匹配与 `new` 标记循环更新。  
- **P2**：统计与全景透视，指标可回溯到条目。  
- **P3**：多维逻辑检索；**二层检索**与「AI 标签字段 / 用户原标签」解耦，便于清理与迁移。  
- **P0**：展示、清理、列定制等基础能力。

#### **面试官提问 1**：「二层检索」解决什么矛盾？  
**考核点**：是否理解解耦、可控清理与进阶检索，不是背名词。  
**高分标准**：说清楚「AI 结构化字段」与「用户既有标签体系」分层的用途与代价。  
**面试回答（总分）**：  
总：解决「要 AI 提取」又要「用户能自主删改原体系」的张力。  
分：第一层用结构化标签支撑精准组合查询；第二层保留用户原生标签不受破坏；这样既提高查准，又不把元数据绑死在一套自动字段上。

#### **面试官提问 2**：理论标签怎么防幻觉、怎么迭代？  
**考核点**：标准库、人工核对、标记策略是否闭环。  
**高分标准**：理论库匹配 → `new` → 核对 → 入库，承认人不能识别所有理论但流程可兜底。  
**面试回答（总分）**：  
总：不靠「模型一次说对」，而靠**标准库 + 异常标记 + 人验**。  
分：第一，优先与标准理论库对齐；第二，新候选打 `new` 提醒核对；第三，确认后纳入库，形成循环；第四，白盒可随时改标签。

#### **面试官提问 3**：标签错了谁负责？你怎么保证最终可用？  
**考核点**：人机分工、学术严谨性、可追溯。  
**高分标准**：AI 出草案，人对关键字段负责；错误可修、可留痕。  
**面试回答（总分）**：  
总：**我对流程与机制负责，不为单次模型输出「背锅」**——用白盒与溯源把结果拉回到可验收状态。  
分：第一，所有标签在 Zotero 可改；第二，综述与对话尽量基于已治理标签而非裸全文；第三，交互回答带来源锚点，便于回原文核对。

---

### 主线 C：生成产出、工程落地与边界（P4–P6 + 路由与推广）

#### **你做了什么**

- **P4**：综述矩阵；以结构化标签组织批量输入（200–300 篇量级指**治理后的输入策略**，非单窗口塞全文）。  
- **P5**：基于标签的引用匹配（精准 / 穷尽等）。  
- **P6**：问答与对话，`[number]` 锚点与溯源；与 Obsidian 等双向链路。  
- **工程与成本**：JavaScript + 大模型 API；多模型路由；版本迭代（如 119 个版本文件量级）；推广路径为种子用户 → 迭代 → 部署培训。

#### **面试官提问 1**：RAG / 综述的输入到底是什么？怎么抑幻觉？  
**考核点**：是否理解 **GIGO** 与「先治理后生产」，不单说「我用了 RAG」。  
**高分标准**：综述输入以结构化中间表示为主，辅以锚点约束。  
**面试回答（总分）**：  
总：**先治理再生成**——直接向全文要综述时幻觉和松散最明显；用好 P1 标签再进 P4，本质是提高信噪比。  
分：第一，输入侧是标签与矩阵化中间件，不是百万字堆进上下文；第二，输出侧要求证据与来源；第三，关键句亦可回原文校验。

#### **面试官提问 2**：多模型怎么分工？成本怎么控？  
**考核点**：任务分层、路由逻辑、是否只会报模型名。  
**高分标准**：按任务难度与质量敏感路由；API 与批处理；自述降本数字带前提（见防守表）。  
**面试回答（总分）**：  
总：**按任务配模型**，而不是一刀切开最强模型。  
分：第一，标签解构、综述、强溯源对话等对质量敏感的路径用更强模型；第二，标准化清理等用更经济组合；第三，批处理与缓存思路降低重复消耗；第四，「约 70%」等必须与基线口径一致。

#### **面试官提问 3**：数据安全、隐私、商业化你怎么说？  
**考核点**：科研数据敏感性、本地与传输边界、诚实描述项目阶段。  
**高分标准**：本地优先、最小传输、私有化习惯；未商业化则如实说。  
**面试回答（总分）**：  
总：文献与未发表材料敏感，我把**本地治理与最小外发**当底线。  
分：第一，核心数据与条目在本地 Zotero；第二，API 只传必要文本片段，策略因课题组而异；第三，私有化部署时明确边界；第四，项目定位如实（研究/开源脚本等），不强装 SaaS 规模。

#### **面试官提问 4（补位·岗位迁移）**：如何把这套方法迁移到企业知识库或「创新方法」相关岗位？

**考核点**：能否抽象三层「标注—检索—再生产」，而非复述 Zotero。  
**高分标准**：与企业案例库/方法库建设同构表述；合规边界点到为止。  
**面试回答（总分）**：  
总：与「案例库、方法库」建设同构。  
分：第一，先定义可录入字段与模板；第二，多维检索与用量/覆盖看板；第三，基于沉淀生成培训材料与推荐；第四，合规场景下再谈 AI 辅助标注。

## 三、其他面试问题预测以及回答

> 说明：本模块承接**第二节三条主线未单独展开**、但业务面仍会追的结构化问题。

### Q1：你没有计算机科班背景，为什么能做完？

**考核点**：学习路径、核心能力是否在工程细节还是架构与规范。  
**高分标准**：脚本为实现手段；主战场是数据规范与验证。  
**面试回答（总分）**：  
总：我的强项在**架构、协议与迭代**，编码是把设计落地。  
分：第一，本科硕士阶段有 Python 与数据分析训练，迁移到 Zotero 的 JS 语境；第二，复杂度在 Prompt 与字段设计，不在炫语法；第三，用版本与模块化控制可维护性，承认边界会补工程债。

### Q2：如果重做，你会改变什么？

**考核点**：复盘深度、指标意识、用户共情。  
**面试回答（总分）**：  
总：**更早做系统化用户实验与指标定义**。  
分：第一，部分迭代早前依赖个人痛点多于抽样验证；第二，准确率、时延、成本应更早成体系监测；第三，把定性反馈改成可对比实验，会减少走弯路。

### Q3：你怎么处理「研究判断」与「合规 / 安全」边界？

**考核点**：与光大 Q4 同类——信息能否用、能否对外说、角色边界。  
**面试回答（总分）**：  
总：可披露、可追溯、最小必要外发。  
分：第一，不拿无法复核的私密传闻当论据；第二，对外陈述区分事实、假设与模型推断；第三，部署时与课题组对齐安全要求；第四，不过度承诺「模型从不犯错」，强调人机协同。

### Q4：这个项目更像「产品」还是「科研工具」？

**考核点**：自我定位清晰，不夸大商业阶段。  
**面试回答（总分）**：  
总：**知识工程向的产品化实践**，有 PRD、指标与迭代，但不等同于成熟商业 SLA。  
分：第一，用产品方法管理需求与版本；第二，当前主场景是科研赋能与实验室采纳；第三，若走向更大规模商业化会另需团队与合规体系——我有意识这个断层。

### Q5：你怎么证明你不是在背名词？

**考核点**：细节与证据链能否随口展开。  
**面试回答（总分）**：  
总：**每个名词对应一条数据流或一次取舍**。  
分：第一，每个模块能说从哪进、从哪出；第二，每个数字能说定义；第三，能讲一个真实迭代（如理论库从哪里来）；第四，愿意承认哪块仍是人工重。
