# BuildZotero —— AI 驱动的知识工程解决方案

## 一、项目定位

BuildZotero 是一套面向科研场景的 AI 知识工程解决方案。

它不是一个网站，不是一个平台，不是一个插件，而是一套依托开源生态的集成式解决方案。具体而言：

- **依托的基础设施**：Zotero（开源文献管理软件）+ Obsidian（笔记与知识管理）+ Cursor（笔记 Agent 终端）
- **依托的插件生态**：Zotero GPT Pro（实现 Zotero 底层数据修改与 AI 配置）、ZotMoov 等（实现与 Obsidian 的工作流连接）
- **核心交付物**：自研的 JavaScript 自动化脚本 + Prompt 工程 + API 编排方案 + 数据流转设计 + 产品文档体系
- **产品形态**：开源脚本集合 + 私有化部署方案

核心思路：不从零造轮子，而是站在 Zotero 和 Obsidian 两个开源生态之上，利用已有插件能力，通过自研脚本和 API 编排，设计了一套贯穿知识输入、整理、输出、查询、管理、沉淀全流程的解决方案。打通了 Zotero（文献阅读）、Obsidian（笔记与知识管理）、Word（文本输出与引用）之间的数据断链与格式转化，让 AI 在知识生产环节不再是功能强但不好用的玩具，而是真正结合场景、需求、工作流实现了价值释放。

---

## 二、核心痛点（3 个维度，4 大壁垒）

通过调研识别出，当前 AI 在科研知识生产端存在 4 大核心壁垒，按知识生产的 3 个阶段归类如下：

### 维度一：数据检索与查找阶段

- **多维复合检索缺失**：传统工具仅支持标题、作者等元数据检索，无法基于"研究方法 + 核心变量 + 样本特征"进行复合逻辑筛选，检索过程极度依赖记忆
- **理论查找与验证断层**：学术理论处于知识前沿，通用 AI 难以穷尽，缺乏结构化的理论库机制，检索结果难以回溯至原文证据点
- **全局态势感知模糊**：面对大规模文献库，无法快速掌握整体分布趋势、方法论占比、变量关联情况

### 维度二：内容生成与输出阶段

- **大规模文献处理瓶颈**：受限于模型上下文窗口，现有工具难以一次性处理 200-300 篇文献的横向对比；通用 RAG 因缺乏学术结构化处理，综述逻辑松散
- **AI 幻觉与证据链缺失**：生成内容存在幻觉风险，缺乏证据溯源机制，用户无法验证综述中的观点是否真实源自文献库
- **引用与格式化成本高**：AI 生成内容与用户个人文献库脱节，引用匹配不精准，需手动调整复杂的学术格式

### 维度三：知识管理与积累阶段

- **知识资产化能力缺失**：传统的点状会话模式导致 AI 提取的信息随 Session 结束而损耗，Token 投入无法转化为长期资产
- **跨文献批量分析能力真空**：需要对数十篇文献进行核心指标、研究局限性或未来方向的批量统计时，仍需大量手动操作
- **工作流孤岛效应**：Zotero、Obsidian 与 AI 分析工具相互独立，交互笔记无法自动关联原文献，知识碎片化

### 归纳为 4 大技术壁垒

1. **AI 上下文长度限制** —— 无法处理大规模文献集
2. **AI 幻觉不可控** —— 结论无法验证、来源无法回溯
3. **AI 输出不稳定** —— 格式不统一，无法直接用于生产
4. **AI 与工作流/工具流断裂** —— 数据在 Zotero / Obsidian / Word 间无法自动化流转

## 三、产品架构（7 个模块，52 个功能点）

系统采用模块化架构，共 7 个核心模块，33 个独立功能，52 个功能点（含版本变体），覆盖科研文献管理全流程：

```
BuildZotero
├── P0: 基础操作模块 (4 个功能)     — 标签展示/清理/删除、标题清理
├── P1: 文献标签解构模块 (7 个功能)  — 核心模块，7 维标签提取（主题/方法/样本/理论/结论/变量/条目细节）
├── P2: 文献标签统计模块 (3 个功能)  — 条目统计、标签统计、自定义统计
├── P3: 文献自定义筛选模块 (4 个功能) — 多维逻辑检索（变量定义/主题分类/控制变量/固定效应）
├── P4: 文献综述模块 (5 个功能)     — 摘要清理、11 维矩阵生成、要点综述、测量综述、研究议程识别
├── P5: 文献引用模块 (1 个功能)     — 自动引用匹配（精准/穷尽/多格式）
└── P6: 交互系统模块 (9 个功能)     — 8 种问答交互（选中文本/标注/剪贴板/PDF/全文/笔记/元数据/知识库）+ 文件上传
```

### 技术规格

- **脚本语言**：JavaScript
- **脚本版本文件总数**：119 个版本迭代文件
- **标签体系**：8 维标注规范 + 11 维分析框架
- **数据格式**：结构化 JSON / Markdown 表格
- **AI 模型配置**：多模型路由策略（DeepSeek V3/R1、Gemini 2.5/3.0 等，按任务复杂度分配以控制成本）
- **产品文档**：15 份（含 PRD、功能清单、用户故事、产品架构、路线图、技术分析等）

## 四、核心设计与功能逻辑

### 4.1 结构化 Schema 提取（P1 模块）

独立设计了一套包含 11 个核心维度的学术分析协议，通过 Prompt 工程将 PDF 全文映射为 Zotero 的标准化标签体系：

- 8 维标签：主题（Item）、方法（sMeth）、样本（Sample）、理论（Theory）、结论（Result）、变量（A1-DV/A2-IV/A3-MO/A4-ME/A5-INV/A6-CV）、条目细节（V1-def/V2-mea/V3-con/V4-fut/V5-sor）
- 每类标签有 V（标准版）、V-R（重处理版）、V-Rnote（笔记优先版）三套版本
- 标签直接写入 Zotero 数据库，用户可随时手动修改（白盒化维护）

### 4.2 理论库循环更新机制

- 理论标签基于标准理论库进行匹配
- AI 识别的新理论加 `Theory/new-` 前缀标记
- 用户核对后纳入标准理论库，形成循环更新
- 解决了"没有人能知道所有理论"的验证困境

### 4.3 来源溯源与一键跳转

- 利用 Zotero 的嵌入式特性，实现 AI 回答与原文的证据挂载
- 所有生成内容带 `[number]` 锚点，支持一键跳转到对应文献段落
- 解决了三个场景的痛点：动态学习时观点无依据、事后总结时来源找不到、写作输出时需要权威引用

### 4.4 自动引用匹配（P5 模块）

- 基于结构化标签进行多维度语义匹配
- 支持精准匹配（1-2 篇最相关）和穷尽式（所有相关）两种策略
- 可在用户撰写的任意内容中，从私有文库秒级匹配权威引用
- 这是目前任何办公软件、阅读软件、笔记软件都做不到的功能

### 4.5 Pipeline 数据流设计

- 采用分层深化逻辑：每个功能端口定义标准化的输入和输出
- 确保下一阶段 AI 应用的输入质量，而不是每个环节都从原料开始
- 通过 Prompt 工程平衡 AI 输出的随机性与格式的确定性
- 通过 API 编排与多模型路由，在质量与成本之间实现平衡

## 七、与主流工具的差异（一句话定位）

> BuildZotero 和 ChatGPT 读 PDF 的区别：ChatGPT 是通用的对话工具，BuildZotero 是结合具体科研场景、工作流和工具链的集成式知识工程方案。它解决的不是"看不懂"的问题，而是"如何让 AI 真正嵌入知识生产的每个环节，把碎片信息转化为可复用、可追溯、可增值的结构化知识资产"的问题。

| 维度 | 通用 AI 工具 | BuildZotero |
|------|-------------|-------------|
| 交互逻辑 | 点状会话，Session 结束知识损耗 | 结构化索引，AI 提取转化为永久标签资产 |
| 检索能力 | 关键词检索，依赖记忆 | 多维逻辑检索，支持"变量 + 方法 + 样本"组合筛选 |
| 处理规模 | 受限于上下文窗口 | 支持 200-300 篇文献同时作为输入 |
| 可维护性 | 黑盒不可改 | 白盒化维护，标签 100% 可修改 |
| 工作流 | 独立工具，信息孤岛 | Zotero + Obsidian + Cursor 双向链接 |
| 理论库 | 无理论库 | 标准理论库匹配 + 新理论标记 + 循环更新 |
| 成本控制 | 单一模型 | 多模型路由策略，按任务分配模型 |
